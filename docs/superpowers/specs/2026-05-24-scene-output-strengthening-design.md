# VoiceInput Pro 场景输出强化设计

目标：让工作台中的不同输入场景对“优化后输出”产生肉眼可见、稳定且可验证的差异，避免当前仅有轻度风格变化、场景辨识度不足的问题。

## 背景

当前系统已经具备场景传参、热词按场景生效、优化 Prompt 按场景分支的基础能力，但整体仍偏“同一套文本优化策略下的轻度改写”。用户虽然能切换场景，但对结果的第一感受往往是“有一点不同，但不够明显”。

本次设计的目标不是继续微调提示词，而是把场景差异提升为产品级特征：

- 同一段原文在不同场景下应表现为不同的输出形态
- 强结构场景必须输出固定结构
- 结果不符合场景契约时，系统应自动纠正
- 差异增强应集中在后端完成，前端尽量少改

## 范围

### 本次纳入

- 重构场景优化 Prompt 生成方式
- 为不同场景定义明确的输出契约
- 为优化结果增加结构校验
- 对不合格结果执行一次自动重写
- 为少量场景增加轻量后处理
- 补充覆盖新行为的后端测试

### 本次不纳入

- 调整 ASR 识别链路本身
- 新增前端复杂交互或可视化配置项
- 引入多模型编排、评分模型或人工审核流
- 改造历史页、统计页的数据结构

## 设计原则

1. 场景差异优先于“温和润色”
2. 结果稳定优先于模型自由发挥
3. 后端约束优先于前端提示
4. 强结构场景必须可程序化验证
5. 自动纠正最多执行一次，控制成本和时延

## 场景输出契约

### 会议纪要

输出必须为固定三段：

- `会议概述：`
- `核心结论：`
- `待办事项：`

约束：

- 三个标题必须全部出现
- `核心结论` 和 `待办事项` 至少各有一条内容
- 待办事项必须是动作句
- 若原文缺少明确任务，输出 `暂无明确待办`

### 工作汇报

输出必须为固定三段：

- `当前进展：`
- `风险问题：`
- `下一步计划：`

约束：

- 三个标题必须全部出现
- 内容必须围绕进展、风险、下一步组织
- 不允许退化成会议纪要结构
- 若没有风险，输出 `当前暂无明显风险`

### Markdown 笔记

输出必须为结构化 Markdown：

- 至少一个一级标题 `#`
- 至少两个二级标题 `##`
- 标题下内容为列表或短段落

建议骨架：

```markdown
# 标题

## 关键信息
- ...

## 细节记录
- ...

## 后续事项
- ...
```

约束：

- 必须是真正可渲染的 Markdown
- 不允许只输出单段正文

### 聊天回复

输出必须为可直接发送的短消息：

- 总句数为 1 到 3 句
- 禁止标题
- 禁止列表
- 禁止总结说明口吻

约束：

- 应保留自然口语感
- 应更像即时沟通，而不是文档或报告

### 正式表达

输出不强制分栏，但必须是正式成文文本。

约束：

- 禁止明显口语填充词
- 禁止列表化表达
- 应适合邮件、通知、说明文等正式场景

### 代码注释

输出不强制分栏，但必须偏技术说明风格。

约束：

- 保留术语、英文缩写、模块名、接口名、变量名等关键信息
- 不能为了自然流畅而改写掉技术词
- 应更接近技术文档、变更说明或代码注释语言

## 架构设计

本次增强以现有处理链路为基础，在优化阶段新增三层约束：

1. `PromptService` 负责生成场景专属 Prompt
2. `SceneOutputValidator` 负责验证输出是否符合场景契约
3. `SceneOutputFormatter` 负责执行轻量后处理

在 `TaskProcessingService` 中串联这三层能力，形成“生成 -> 校验 -> 必要时重写 -> 格式修整 -> 入库”的稳定链路。

## 组件设计

### 1. PromptService 重构

当前 `PromptService` 采用通用模板拼接场景说明。本次改为“场景专属模板”。

建议保留统一入口方法，但内部按场景拆分：

- `buildMeetingMinutesPrompt(...)`
- `buildWorkReportPrompt(...)`
- `buildFormalExpressionPrompt(...)`
- `buildMarkdownNotePrompt(...)`
- `buildCodeCommentPrompt(...)`
- `buildChatReplyPrompt(...)`

每个模板都必须明确：

- 角色定位
- 场景目标
- 输出结构
- 禁止事项
- 返回 JSON 的字段约束

### 2. SceneOutputValidator

新增独立服务，职责是依据 `SceneType` 校验优化结果是否满足场景契约。

建议暴露统一接口：

```text
validate(sceneType, optimizedText, markdown) -> ValidationResult
```

`ValidationResult` 至少包含：

- `passed`
- `reason`
- `retryHint`

示例校验规则：

- 会议纪要：是否存在 3 个固定标题
- 工作汇报：是否存在 3 个固定标题
- Markdown：是否包含 `#` 与 `##`
- 聊天回复：是否为 1 到 3 句，且无标题与列表符号
- 正式表达：是否存在明显列表或强口语痕迹
- 代码注释：是否保留常见技术标识，不作为强失败项但可给出提醒

### 3. SceneOutputFormatter

新增轻量格式化服务，负责在模型结果基本正确时做产品级收尾，而不是重新生成内容。

典型职责：

- 为会议纪要、工作汇报统一列表编号
- 为缺失但允许补默认值的块填入占位句
- 为聊天回复裁剪到最多 3 句
- 为 Markdown 内容补齐基础标题骨架

Formatter 只做小修整，不承担大规模内容改写。

### 4. TaskProcessingService 串联逻辑

现有链路为：

- 识别原文
- 应用热词
- 调用优化模型
- 写回数据库

新链路改为：

1. 识别原文
2. 应用热词
3. 生成场景专属 Prompt
4. 调用优化模型
5. 校验结果
6. 若失败，则基于失败原因执行一次重写
7. 对最终结果做轻量格式化
8. 写回数据库

重写次数上限为 1 次，避免无限重试。

## 数据流

### 主流程

1. 前端提交任务，并携带 `sceneType`
2. 后端读取 `sceneType` 与当前配置快照
3. `PromptService` 生成该场景的专属优化 Prompt
4. 模型返回 `title / summary / optimizedText / markdown / improvement`
5. `SceneOutputValidator` 校验结果是否符合场景契约
6. 若不符合，构造“纠偏 Prompt”并再次调用模型
7. `SceneOutputFormatter` 对合格结果做收尾
8. 将最终结果、热词命中、耗时、成本写入数据库

### 纠偏流程

纠偏 Prompt 不重新描述完整业务上下文，而是聚焦：

- 当前场景
- 首次输出失败原因
- 目标结构
- 禁止保留的错误形态

例如：

- 缺少 `待办事项`
- 误写成普通正文
- Markdown 没有二级标题
- 聊天回复超过 3 句

## 错误处理

### 模型正常返回但结果不合格

- 记录校验失败原因
- 自动触发一次重写
- 若重写后仍不合格，则保留第二次结果并写入日志

这里不将任务直接标记失败，因为多数情况下内容仍可用，只是结构不理想。

### 模型调用异常

- 维持现有异常处理方式
- 任务标记为 `FAILED`
- 错误信息进入 `errorMessage`

### Formatter 无法修整

- 不抛出致命异常
- 保留模型原结果
- 写入 warning 日志

## 测试设计

### PromptService 测试

- 每个场景生成的 Prompt 都包含对应结构要求
- 强结构场景包含禁止事项
- 会议纪要场景不会错误带入工作汇报结构

### SceneOutputValidator 测试

- 每个强结构场景至少一条合格样例
- 每个强结构场景至少一条不合格样例
- 聊天回复句数、标题、列表规则单独覆盖

### TaskProcessingService 流程测试

- 首次输出合格时不触发重写
- 首次输出不合格时触发一次重写
- 重写最多执行一次
- 最终入库内容来自格式化后的结果

### 人工验收样例

准备 5 段固定原文：

- 会议口语
- 项目汇报
- 零散笔记
- 技术说明
- 简短沟通

每段原文分别跑 6 个场景，确认至少以下 4 个场景一眼可区分：

- 会议纪要
- 工作汇报
- Markdown 笔记
- 聊天回复

## 验收标准

1. 同一段原文在 6 个场景中，至少 4 个结果肉眼明显不同
2. 会议纪要、工作汇报、Markdown 笔记、聊天回复必须命中各自固定结构
3. 正式表达不能退化为口语聊天或列表纪要
4. 代码注释场景必须保留关键技术术语
5. 结果不合格时，系统能自动纠偏一次
6. 绝大多数任务第一次即合格，少数任务第二次拉正

## 实施顺序

### 第一阶段：最小可见收益

1. 重构 `PromptService` 为场景专属模板
2. 新增 `SceneOutputValidator`
3. 在 `TaskProcessingService` 中接入一次自动重写

这一阶段完成后，应先验证“场景差异是否明显增强”。

### 第二阶段：稳定性与观感打磨

1. 新增 `SceneOutputFormatter`
2. 为会议纪要、工作汇报统一列表与缺省项
3. 为 Markdown 与聊天回复做轻量收尾

### 第三阶段：测试补齐

1. 补 Prompt 模板测试
2. 补校验器测试
3. 补处理流程测试

## 风险与权衡

### 成本上升

自动重写会带来额外 token 成本与耗时。通过“最多一次重写”控制风险。

### 改写更重

为追求场景差异，结果会比当前版本更偏结构化改写。这是有意接受的产品取舍。

### 短文本模板感增强

短文本进入强结构场景时可能显得模板感较重。通过默认占位句和格式化收尾缓和观感。

### 聊天回复过度加工

聊天回复场景必须在 Prompt、校验、Formatter 三层都保持短句约束，避免退化为“小作文”。

## 推荐结论

推荐按以下组合落地：

- 场景专属 Prompt
- 结构校验
- 最多一次自动重写
- 轻量后处理

这套方案改动集中在后端，收益明显，能够把“场景切换确实生效”提升为“场景差异一眼可见、可演示、可验收”的产品能力。
