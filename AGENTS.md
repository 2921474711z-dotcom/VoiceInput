# VoiceInput Pro 全栈交付开发文档 V2

> 本版为当前唯一有效的开发约束，基于 `frontDesign` 中现有页面重新整理。
> 本文件后续旧版内容全部视为归档内容，不再作为开发和验收依据。

## 1. 文档定位

本文件用于约束 `VoiceInput Pro` 的前端、后端、数据库、中间件、Docker 编排和联调交付标准。

目标不是只把页面画出来，而是把页面上已经出现的入口、按钮、筛选、抽屉、图标操作、详情区、分页、导入导出、配置切换全部做成真实功能，并让数据在页面、接口、数据库、中间件之间完整打通。

本项目统一名称：

`VoiceInput Pro`

## 2. 最高原则

### 2.1 页面上出现的元素必须真实可用

以下元素只要保留在页面上，就必须具备真实行为：

- 按钮
- 图标按钮
- 上传框
- Tab
- 下拉框
- 单选框
- 开关
- 搜索框
- 筛选器
- 排序器
- 分页器
- 详情抽屉或侧栏
- 导入与导出入口
- 清空、删除、恢复默认等危险操作

禁止以下情况：

- 只有静态 UI，没有后端行为
- 点击后无反馈
- 点击后只提示“敬请期待”
- 假保存
- 假导出
- 假删除
- 假筛选
- 假排序
- 假分页
- 假配置切换
- 假图表
- 假统计数据

原则只有一条：

`宁可删掉入口，也不能保留一个无效入口。`

### 2.2 数据必须全链路打通

凡是页面上能看到的数据，都必须能追溯到真实来源，至少打通到以下层级：

- 前端状态
- 后端接口
- 数据库持久化
- 中间件或任务通道
- 文件存储或对象存储（涉及音频、导出文件时）

禁止仅在前端本地写死以下内容：

- 历史记录列表
- 热词列表
- 配置保存结果
- 处理耗时
- 估算成本
- 字数
- 热词命中
- 使用统计
- 样例对比

### 2.3 Docker 一键启动

项目交付必须支持通过 Docker Desktop 启动完整依赖，至少满足：

- `docker compose up -d` 可以启动核心依赖
- 前后端及依赖服务可以联调
- 初始化脚本自动建库建表或自动迁移
- 服务具备健康检查
- 默认数据目录、挂载目录、数据库卷、对象存储目录统一落在 `E:\DockerData`

如果使用本机 Docker Desktop，必须遵守：

- 镜像相关数据、挂载目录、卷映射根目录统一放在 `E:\DockerData\VoiceInputPro`
- 不允许把数据库卷、对象存储目录随意散落到系统盘

如果后端使用 Maven，必须遵守：

- Maven 配置文件目录：`E:\Maven`
- Maven 本地仓库：`E:\maven-repository`

## 3. 技术栈与目录约束

项目目录必须与当前工作区保持同层级结构：

- `frontDesign`：前端视觉参考目录
- `frontend`：前端项目目录
- `backend`：后端项目目录

### 3.1 前后端关系

本项目必须采用前后端分离架构：

- 前端独立启动
- 后端独立启动
- 前端通过 HTTP API 调用后端
- 前端页面数据必须来自后端接口

### 3.2 前端技术栈

前端技术栈允许开发阶段自主选择，但必须满足以下要求：

- 适合实现复杂工作台、表格、详情侧栏、筛选、分页和配置页面
- 支持组件化开发
- 支持状态管理和 API 联动
- 支持 Docker 化部署

当前推荐栈：

- `React`
- `TypeScript`
- `Vite`

如果后续改用 Vue 等其他前端方案，也必须满足：

- 页面结构不变
- 交互闭环不变
- 数据联动不变
- Docker 启动方式不变

### 3.3 后端技术栈

后端主服务语言以 `Java` 为主，必要时可配合 `Python` 处理专项能力，但核心业务必须由 Java 主导实现。

后端必须至少覆盖以下职责：

- REST API
- 业务编排
- 任务处理
- 数据库存取
- 配置管理
- 统计聚合
- 导入导出
- 中间件接入

当前推荐栈：

- `Java 17+`
- `Spring Boot 3`
- `Spring Web`
- `Spring Data JPA` 或等价持久层方案
- `Flyway` 或等价数据库迁移方案

### 3.4 Python 使用边界

如果项目中使用 Python，只允许用于以下增强场景：

- 音频预处理
- 转码
- OCR 或特殊文本清洗
- 第三方 AI 能力桥接
- 离线数据处理脚本

Python 不能替代 Java 主后端承担以下核心职责：

- 主 API 服务
- 主业务编排
- 主数据库访问层
- 主配置管理
- 主历史记录服务
- 主热词管理服务

### 3.5 基础设施技术

基础设施推荐如下：

- 数据库：`PostgreSQL` 或 `MySQL`
- 缓存与任务通道：`Redis`
- 对象存储：`MinIO`
- 容器编排：`Docker Compose`

### 3.6 README 必须写清技术栈

最终 `README` 中必须明确写清：

- 前端用了什么技术栈
- 后端用了什么技术栈
- 中间件用了什么
- 大模型和 ASR 接了什么协议

## 5. 后端能力与数据链路要求

## 5.1 必须打通的业务闭环

至少完成以下 5 条闭环：

1. 上传闭环：上传音频 -> 存储文件 -> 创建任务 -> 识别 -> 优化 -> 展示结果
2. 历史闭环：保存历史 -> 查询列表 -> 打开详情 -> 再次优化 -> 生成新版本
3. 热词闭环：新增热词 -> 启用热词 -> 参与处理 -> 结果体现修正 -> 统计热词命中
4. 配置闭环：修改配置 -> 保存配置 -> 后续处理生效 -> 统计结果变化
5. 统计闭环：任务记录 -> 聚合分析 -> 图表展示 -> 文案总结

## 5.2 建议的数据流

建议采用如下链路：

1. 前端上传音频
2. 后端接收并写入对象存储或挂载目录
3. 后端创建任务记录写入数据库
4. 后端将任务投递到 Redis Stream、RabbitMQ 或等价任务中间件
5. Worker 执行识别、热词修正、文本优化
6. Worker 将结果、统计数据、热词命中、版本信息写回数据库
7. 前端通过轮询或推送获取任务状态与结果

要求：

- 如果页面显示“处理中”，后端必须有真实处理中状态
- 如果页面显示“处理失败”，后端必须有真实失败原因
- 不允许前端自己拼一个成功结果冒充异步任务

## 6. 数据库与中间件要求

## 6.1 数据库

必须接数据库，不能只靠前端本地存储。

建议至少包含以下核心实体：

- `processing_tasks`：处理任务主表
- `task_results`：任务结果表
- `task_versions`：再次优化版本表
- `hotword_categories`：热词分类表
- `hotwords`：热词表
- `hotword_samples`：热词样例表
- `model_configs`：当前配置表
- `config_templates`：配置模板表
- `exports`：导出记录表
- `usage_stats_daily`：按天聚合统计表或视图

要求：

- 任务、热词、配置、模板、导出记录必须持久化
- 历史列表、统计报表必须可以通过数据库重新查询恢复

## 6.2 中间件

以下依赖建议作为最小必备：

- MySQL 8.x 或 PostgreSQL 15+
- Redis 7.x
- MinIO

异步任务二选一：

- Redis Stream
- RabbitMQ

要求：

- Redis 至少承担缓存、任务状态、幂等控制或队列能力中的一部分
- MinIO 或等价对象存储至少承担音频文件、导出文件存储
- 热词启停、配置更新后，如果存在缓存，必须有缓存失效或刷新机制

## 2. 产品目标

本项目不是做一个“录音转文字”小工具，而是做一个面向办公与技术场景的智能语音输入工作台。

评审必须能够快速看懂三件事：

1. 这不是单纯 ASR，而是“可直接产出可用文本”的输入产品。
2. 这不是静态设计稿，而是可操作、可验证、可演示的完整原型。
3. 这套方案对真实场景有价值，尤其适合会议纪要、工作汇报、技术表达和专业术语输入。

## 3. 核心竞争力

项目必须同时具备以下四个竞争力支点，缺一不可：

### 3.1 场景化输入

- 工作汇报
- 会议纪要
- 正式表达
- Markdown 笔记
- 代码注释
- 聊天回复

每个场景都必须真实影响输出结果，例如：

- 工作汇报：输出分点总结
- 会议纪要：输出议题、结论、待办
- 正式表达：减少口语化表达
- Markdown 笔记：输出 Markdown 结构
- 代码注释：保留技术术语并压低误识别
- 聊天回复：保持简洁自然

### 3.2 识别后优化

系统必须同时展示：

- 原始识别文本
- 优化后文本

- 断句

- 冗余清理
- 按场景结构化整理

### 3.3 热词修正

系统必须支持专业术语和常见误识别修正，这是项目的核心差异化能力。

热词必须真实参与处理流程，并在结果中体现修正效果。

### 3.4 评测反馈

- 成本

- 优化效果样本对比

## 补充交付原则（2026-05-23）

整个项目不允许只为了“当前这台机器能跑”而做一次性适配。

必须优先满足以下原则：

- 别人拿到项目后，最多只需要按 `README` 配置环境变量、准备依赖、执行标准启动命令，就可以启动和联调
- 不能依赖开发者本机手工安装但项目中没有声明的工具、脚本、路径或服务
- 如果需要额外运行时能力，优先放入 Docker 镜像、`docker compose`、项目依赖、初始化脚本或明确的安装步骤中
- 不允许把“必须在作者电脑上额外做的动作”当成默认前提
- 本机专用配置只能放在本地覆盖层或示例外部，不能污染通用交付配置

开发时默认优先选择：

- `项目自带/容器内可复现` 的方案
- `README 可说明、他人可复现` 的方案

而不是：

- `依赖作者本机现成环境` 的方案
- `只有当前机器能直接成功` 的方案
